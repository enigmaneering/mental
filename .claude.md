# Claude Code Instructions

## Git Workflow Rules

**IMPORTANT**: Do NOT automatically commit, push, or create/push tags.

- ❌ Auto-commit: **DISABLED**
- ❌ Auto-push: **DISABLED**
- ❌ Auto-tag: **DISABLED**

Only create commits, push to remote, or create/push tags when **explicitly requested** by the user in the current conversation.

## Repository Overview

This is `enigmatic` - The Enigmaneering Guild CLI Tool (`e`).

A command-line interface that provides tools and utilities for The Enigmaneering Guild's ecosystem, including the `fetch` command for downloading shader compilation toolchains from the redistributables repository.

## Repository Structure

### Core Files
- `main.go` - CLI entry point with subcommand routing
- `go.mod` - Module definition: `git.enigmaneering.org/enigmatic`
- `gpu/vendor.go` - Library for downloading redistributables from `enigmaneering/redistributables`
- `gpu/README-gpu.md` - Documentation for the GPU library
- `.github/workflows/build-fetch.yml` - Workflow named "Build e" that builds `e` binaries for all platforms
- `README.md` - User-facing documentation
- `license` - The Enigmaneering License (MIT-based)

### Commands
- `e fetch` - Download shader compilation toolchain binaries
- `e help` - Show help message

### Build System
- Builds binary named `e` (not `enigmatic` or `fetch`)
- Supports all platforms: darwin-arm64, darwin-amd64, linux-amd64, linux-arm64, windows-amd64, windows-arm64
- Releases created via GitHub workflow (manual trigger only)
- Version tags are simple: `v0.0.42` (no prefix)

## Development Workflow

### Making Changes
1. Make code changes
2. Test locally: `go build -o e && ./e fetch -help`
3. Only commit/push when user explicitly requests it
4. **Never automatically create or push version tags**

### What NOT to Commit
- ❌ Compiled binaries (`e`, `e-*-*`)
- ❌ Downloaded external libraries in test directories
- ❌ Temporary files

### What TO Track
- ✅ Source code (`main.go`, `gpu/*.go`)
- ✅ Documentation (`README.md`, `gpu/README-gpu.md`)
- ✅ Workflows (`.github/workflows/*.yml`)
- ✅ `go.mod` and `go.sum`
- ✅ License file

## Creating Releases

**IMPORTANT**: Never automatically create or push tags. Releases must be manually triggered.

To create a release:
1. User commits all changes
2. User manually creates and pushes tag: `git tag v0.0.X && git push origin v0.0.X`
3. User manually triggers "Build e" workflow in GitHub Actions with version input

## Module Path

- Vanity URL: `git.enigmaneering.org/enigmatic`
- Maps to: `https://github.com/enigmaneering/enigmatic`
- Import as: `import "git.enigmaneering.org/enigmatic/gpu"`

## Related Repositories

- `enigmaneering/redistributables` - Pre-built shader compilation binaries
  - The `e fetch` command downloads from this repo's releases
- `enigmaneering/life` - GPU device abstraction library
  - Imports `git.enigmaneering.org/enigmatic/gpu` to auto-download toolchain

## Common Tasks

### Build Locally
```bash
go build -o e
./e help
./e fetch
```

### Test the fetch command
```bash
./e fetch -version v0.0.42
./e fetch -dir /tmp/test-shaders
```

### Update redistributables reference
The `gpu/vendor.go` file has:
```go
const githubRepo = "enigmaneering/redistributables"
```

This should always point to the redistributables repository (not external or enigmatic).

## Testing Philosophy

Manual testing of CLI commands:

- Test flag parsing
- Test help output is clear and accurate
- Test downloading from redistributables releases
